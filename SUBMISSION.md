# LFPY 제출용 데이터 활용 설명

## 1. 서비스 개요
LFPY(“Let’s Find the Place You want”)는 사용자가 설문(12문항) 또는 텍스트 입력을 통해 선호 조건을 전달하면, **전국 고등학교 후보군**을 필터링하고 **점수 기반 추천 TOP 3**를 제공하는 학생용 학교 추천 서비스입니다.  
대회 요구사항인 “교육 공공데이터 활용”을 위해, 본 서비스는 다음의 공개 데이터 소스를 조합합니다.

**중요:** 이 서비스의 학교 추천 판단은 원칙적으로 **공공데이터만으로** 이루어집니다.  
즉, 학교 후보군 구성, 필터링, 추천 점수 산정, 상세 표시의 사실 확인 단계에서 **교육 공공데이터를 중심 데이터로 사용**합니다.

## 2. 사용한 교육 공공데이터(핵심)

### 2.1 전국 고등학교 목록/기본 속성(정적 베이스 데이터)

| 항목 | 내용 |
|---|---|
| 데이터 원천 | 사용자 제공 전국 고등학교 `.xls` |
| 파일명 | `2416264D5678F46D16.xls` |
| 사용 목적 | 후보 학교 목록 생성, 필터링, 점수 계산의 기준 데이터 |
| 활용 범위 | 학교명, 지역, 행정구, 설립구분, 학교유형, 주소, 홈페이지 등 공개 정보 |

- **역할(앱에서의 사용 목적)**  
  1) 추천 대상 후보군을 구성합니다. (전국 학교 목록)  
  2) 검색/필터 UI를 구성합니다. (시도/행정구/유형/설립구분 등)  
  3) 추천 알고리즘의 “매칭 근거”를 만듭니다.  
     - 각 학교의 이름/주소/유형/설립구분/태그/키워드를 정규화한 `searchText`, `scoreText`를 생성
     - 사용자가 입력한 설문 조건과 키워드 포함 여부를 기반으로 점수를 계산
- **앱 내 처리 흐름(논리)**  
  1) `.xls` → `nationalHighSchools.js`의 `NATIONAL_HIGH_SCHOOLS` 배열로 변환(정제 포함)  
  2) 앱 실행 시 `SCHOOLS`로 로딩  
  3) 각 학교 객체에 대해 `searchText`, `scoreText`, `homepageUrl` 등을 전처리하여 사용
- **필드 활용(추천/검색 로직 기준)**  
  - `학교명(name)`
  - `시도/지역(sido/city/region)`
  - `행정구(district)`
  - `설립구분(founding: 공립/사립)`
  - `학교유형(type: 일반고/외국어고/국제고/과학고/예술고/특성화고 등)`
  - `주소(address)`, `전화/팩스(있을 경우)`, `홈페이지(homepageUrl)`
  - (있을 경우) `tags`, `keywords` → 추천 점수 및 추천 이유 구성

### 2.2 NEIS Open API(최신 학교 정보 보정)
- **데이터 원천(교육행정 공개 API)**: NEIS Open API의 `schoolInfo` 엔드포인트
- **역할(앱에서의 사용 목적)**  
  1) 정적 베이스 데이터가 최신 정보와 다를 수 있으므로,  
  2) 사용자가 실제로 보는 화면(추천 TOP 3 / 상세 보기)에 한해서 **NEIS로 최신 정보를 조회**해 보정합니다.
- **앱 내 처리 흐름(논리)**  
  1) 추천 후보를 점수로 정렬  
  2) TOP 후보를 순차적으로 NEIS 조회 (`fetchLiveSchoolData`)  
  3) NEIS 응답에서 학교 정보가 확인되지 않으면 해당 후보는 제외(추천/상세에서 스킵)  
  4) NEIS로 확인된 학교에 대해 `homepage`, `주소`, `지역` 등을 병합(`mergeLiveSchoolData`)하여 표시
- **코드 수준에서의 핵심 포인트**
  - 학교명(`SCHUL_NM`)을 기본 키로 조회
  - 가능하면 교육청 코드(`ATPT_OFCDC_SC_CODE`)를 함께 사용하도록 오피스 코드 매핑 로직을 적용
  - 캐싱(`LIVE_SCHOOL_CACHE`)으로 동일 학교 재조회 비용을 줄임

### 2.3 교육 공공데이터 활용을 “꼭” 넣은 이유
- 공공데이터 없이도 단순 UI는 만들 수 있지만, 이 프로젝트는 **학교 추천의 판단 근거 자체를 공공데이터로 증명**해야 합니다.
- 따라서 본 서비스는 다음 순서를 따릅니다.
  1) **교육 공공데이터로 학교 후보를 구성**
  2) **교육 공공데이터의 공개 필드로 검색/필터/추천 점수 계산**
  3) **교육 공공데이터(NEIS)로 최신성 검증**
  4) **검증되지 않은 학교는 표시에서 제외**

## 3. 교육 공공데이터 활용의 “구체성/체계성” 설명(제출 관점)
본 서비스는 교육 공공데이터를 단순 표시가 아니라 **데이터 처리 파이프라인(필터링→전처리→점수화→화면 보정)**으로 활용합니다.

1) **정적 베이스 데이터(전국 목록)**  
   - 대규모 후보군을 만들고(검색/필터 가능) 추천 점수의 토대를 구축합니다.
2) **전처리(정규화/검색 텍스트 생성)**  
   - 사용자가 입력하는 설문 조건(텍스트/선택지)을 `scoreText`와 비교 가능하도록 통일된 형태로 변환합니다.
3) **추천 알고리즘(가중치 기반 점수화)**  
   - 유형/설립구분/키워드/직업 키워드 등 포함 여부를 항목별로 가중치 점수로 합산합니다.
4) **NEIS Open API(최신성 보정)**  
   - 사용자가 실제로 “상위 추천” 및 “상세 보기”를 눌렀을 때만 NEIS를 호출하여 최신 정보를 화면에 반영합니다.
5) **비운영 학교/정보 부재 대응(제외 로직)**  
   - NEIS 조회 결과가 없으면 추천/상세에서 스킵하여, 사용자에게 불필요한 비정확 정보를 제공하지 않도록 합니다.

## 4. 위치 기반 자동 필터링(보조 데이터)
학교 추천의 “지역 입력 편의”를 위해,
- 브라우저 `Geolocation API`로 좌표를 읽고
- OpenStreetMap Nominatim 역지오코딩으로 시도/행정구를 추정
- 추정된 시도/행정구를 검색 필터에 자동 반영

이 부분은 교육 공공데이터 그 자체라기보다는, 사용자가 지역을 직접 입력하지 않아도 추천을 빠르게 좁히게 해주는 공개 지리 데이터 활용입니다.

## 5. 개인정보/권한 처리(요약)
- 사용자의 위치를 읽어 자동 필터링에만 사용합니다.
- 서비스는 Firebase 트래킹/관리자 모드 등은 제거되어, 학생용 추천 흐름에 필요한 최소 기능만 유지합니다.

## 6. 제출용 한 줄 요약
LFPY는 **전국 고등학교 공공데이터(.xls 정제본)**로 후보군/추천 근거를 만들고, **NEIS Open API(schoolInfo)**로 상위 추천 및 상세 화면에 대해 최신성을 보정하여 “필터링+점수 추천+최신 확인”의 체계적 데이터 활용을 구현했습니다.
